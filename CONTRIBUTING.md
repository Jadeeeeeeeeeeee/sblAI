# 🤝 Contributing to SBL AI

Thanks for your interest in contributing to **SBL AI (Science-Based Lifting AI)**.  
The goal of this project is to build a **clean, reliable, and evidence-based fitness tool** that helps people train smarter.

---

## 📌 Core Principles

All contributions should follow these core principles:

- **Science-first**: Any fitness-related logic must be based on peer-reviewed research or widely accepted evidence-based training principles.
- **Practical over theoretical**: Prioritize what actually helps lifters in real training situations.
- **Simplicity matters**: Keep solutions clean and avoid unnecessary complexity.
- **Consistency**: Match existing code style, structure, and patterns.
- **No bro-science**: Avoid unsupported claims or purely anecdotal fitness advice.

---

## 🧑‍💻 Code Standards

- Write clean, readable code
- Use meaningful variable and function names
- Keep functions small and focused
- Avoid duplicated logic (DRY principle)
- Add comments only where logic is not obvious
- Avoid introducing breaking changes without clear explanation

---

## 🧪 Fitness / Training Logic Contributions

If you're working on anything related to programming workouts, progression, volume, or training systems:

- Base your logic on:
  - Peer-reviewed studies
  - Established strength & hypertrophy principles
  - Evidence-based coaching frameworks
- Clearly state assumptions (e.g. hypertrophy-focused vs strength-focused)
- Avoid one-size-fits-all programming
- Consider:
  - Beginner vs advanced lifters
  - Recovery and fatigue
  - Progressive overload principles

---

## 🐛 Bug Reports

When reporting a bug, please include:

- What happened
- What you expected to happen
- Steps to reproduce
- Device / OS / browser (if relevant)
- Screenshots or logs if available

---

## ✨ Feature Requests

Good feature requests should include:

- A clear description of the feature
- Why it improves the product
- A real-world use case
- (Optional) UI or logic suggestion

---

## 🔀 Pull Request Guidelines

Before submitting a PR:

- Ensure your code runs correctly
- Test your changes locally
- Keep PRs focused (one feature or fix per PR)
- Write a clear title and description
- Link related issues if applicable

---

## 🚫 What Will NOT Be Accepted

- Unsupported fitness claims
- Overly complex or unnecessary features
- Breaking changes without explanation
- Unsafe or misleading training recommendations

---

## 💬 Communication Style

- Be respectful and constructive
- Ask questions when unsure
- Keep discussions technical and focused
- Avoid assumptions—clarify instead

---

## 🙌 Final Note

Good contributions improve both the **codebase and the training quality of the app**.  
If you're unsure about something, open a discussion before building it.

Thanks for helping improve SBL AI 💪
